# 🌐 Peer-to-Peer (P2P) Architecture 📡⚡

> _A decentralized networking model where peers both consume and provide services — turning every node into both client and server._

---

# 💠 Core Intuition — Why P2P Exists

Traditional client-server systems depend heavily on centralized infrastructure.

Problem:

```text id="jlwmh3"
More users = more server load
```

P2P flips the model:

```text id="k3g26l"
More users = more total capacity
```

Every new peer contributes:

- 📤 Upload bandwidth
- 💾 Storage
- ⚡ Distribution capacity

---

# 🧠 What is Peer-to-Peer Architecture?

## 💠 Definition

A P2P system has:

- ❌ No permanently active central server
- 🔄 Direct peer-to-peer communication
- ⚡ Shared resource contribution

Each node acts as:

| Role   | Function            |
| ------ | ------------------- |
| Client | Downloads resources |
| Server | Uploads resources   |

---

# 🧪 Key Characteristics of P2P

| Feature              | Description                           |
| -------------------- | ------------------------------------- |
| Self-scalability     | More peers increase system capacity   |
| Decentralization     | No single bottleneck                  |
| Dynamic peers        | Nodes frequently join/leave           |
| IP variability       | Peers may constantly change addresses |
| Distributed workload | Traffic shared among peers            |

---

## ⚠️ Main Challenge

P2P systems are operationally complex because peers are unstable.

Examples:

- 🔄 Churn (frequent disconnects)
- 🌐 NAT traversal
- 📡 Dynamic IP changes
- 🔒 Trust/security concerns

---

# ⚔️ Client-Server vs P2P File Distribution

# 💠 Distribution Problem

A server wants to distribute:

```text id="pjlwm"
File size = F
```

to:

```text id="dr4y11"
N peers
```

---

# 🧪 System Variables

| Symbol  | Meaning                    |
| ------- | -------------------------- |
| `F`     | File size                  |
| `N`     | Number of peers            |
| `u_s`   | Server upload rate         |
| `u_i`   | Peer upload rate           |
| `d_i`   | Peer download rate         |
| `d_min` | Slowest peer download rate |

---

# 🏢 Client-Server Distribution

## 💠 How It Works

The central server uploads every copy itself.

---

## 🔄 Workflow

```text id="v3w6ul"
Server
 ├──► Peer 1
 ├──► Peer 2
 ├──► Peer 3
 └──► Peer N
```

---

# 🧪 Distribution Time Formula

Server must upload:

```text id="5l0cxf"
N copies of the file
```

Time lower bound:
D_CS >= max( (N \* F) / u_s, F / d_min )

---

## ⚡ Interpretation

| Component  | Meaning                            |
| ---------- | ---------------------------------- |
| `NF / us`  | Server upload bottleneck           |
| `F / dmin` | Slowest client download bottleneck |

---

## ⚠️ Scalability Problem

As users increase:

```text id="r5ldsq"
Server workload grows linearly
```

Meaning:

- More traffic
- Higher infrastructure cost
- Central bottleneck risk

---

# 🌐 P2P Distribution

## 💠 Core Idea

Peers help distribute the file to each other.

The server uploads only part of the workload.

---

## 🔄 Workflow

```text id="69jg92"
Peer A ◄────► Peer B
   ▲            │
   │            ▼
Server ◄────► Peer C
```

---

# 🧪 P2P Distribution Formula

Lower bound:

D_P2P >= max( F / u_s, F / d_min, (N \* F) / (us + sum(u_i)) )

---

# ⚡ Why P2P Scales Better

As peers increase:

- Total demand increases ✅
- Total upload capacity also increases ✅

This creates:

```text id="ymv6fd"
Self-scalability
```

---

## 🧠 Key Insight

Client-server:

```text id="ldu4c3"
More users = more server stress
```

P2P:

```text id="b3fmyk"
More users = more system power
```

---

# ⚖️ Client-Server vs P2P Comparison

| Feature               | Client-Server | P2P     |
| --------------------- | ------------- | ------- |
| Scalability           | Limited       | High    |
| Central bottleneck    | Yes           | Minimal |
| Infrastructure cost   | High          | Lower   |
| Management simplicity | Easier        | Harder  |
| Fault tolerance       | Lower         | Higher  |
| Dependency on server  | Strong        | Weak    |

# 🧲 BitTorrent Architecture

# 💠 What is BitTorrent?

BitTorrent is a large-scale P2P file-sharing protocol.

---

## 🧪 File Chunking

Files are divided into smaller chunks.

Example:

```text id="klmw1l"
256 KB chunks
```

---

## ⚡ Why Chunking Matters

Benefits:

- Parallel downloading
- Faster distribution
- Better resilience
- Efficient sharing

---

# 🌍 Torrent Ecosystem

## 💠 Main Components

| Component | Purpose                     |
| --------- | --------------------------- |
| Tracker   | Maintains active peer list  |
| Torrent   | Group of peers sharing file |
| Peer      | Downloads/uploads chunks    |

---

## 🔄 Peer Join Process

```text id="mxq4xd"
1. Peer joins torrent
2. Contacts tracker
3. Gets peer list
4. Connects to neighbors
5. Starts exchanging chunks
```

---

# 🔄 Churn in P2P Networks

## 💠 Churn Definition

Peers constantly:

- Join
- Disconnect
- Reconnect

---

## ⚠️ Impact

Churn creates:

- Unstable topology
- Routing complexity
- Availability uncertainty

---

# 🧪 BitTorrent Download Strategy

# 💠 Rarest-First Algorithm

Peers prioritize downloading rare chunks first.

---

## ⚡ Why It Works

Prevents:

```text id="vr98ep"
Rare chunks from disappearing
```

Improves:

- Availability
- Redundancy
- Distribution fairness

---

# 🔄 Chunk Request Workflow

```text id="e3b2hf"
1. Ask neighbors for chunk lists
2. Identify missing chunks
3. Request rarest chunks first
```

---

# ⚔️ Tit-for-Tat Mechanism

# 💠 Core Idea

Peers reward peers that upload quickly to them.

---

## 🧪 Behavior

Alice uploads mainly to:

```text id="yyec1v"
Top 4 peers sending her data fastest
```

Others are:

```text id="pqjz2g"
Choked
```

---

# ⚡ Optimistic Unchoking

Every 30 seconds:

```text id="i5v7y5"
Random peer temporarily receives upload access
```

Purpose:

- Discover faster peers
- Allow newcomers to participate
- Improve fairness

---

# 🧠 Tit-for-Tat Dynamics

## 🔄 Example Flow

```text id="8kc9kq"
1. Alice unchokes Bob
2. Bob uploads faster to Alice
3. Alice prioritizes Bob
4. Mutual high-speed exchange forms
```

---

# ⚡ System Impact

Tit-for-tat incentivizes cooperation and discourages freeloading.
